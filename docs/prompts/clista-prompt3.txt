TOPIC: Is ClisTa Protocol a real, generalizable protocol layer for accountable agentic work, or is it adequately solved by one strong prompt plus one markdown transfer-state document?
CORE CLISTA LAW: Conversation is input. Reasoning state is output.
OPERATING MODEL: Explore in threads. Project into state. Verify against artifacts. Act only after alignment.
Spawn exactly four teammates:
PROTOCOL_ADVOCATE: Argues that ClisTa should become a full protocol/runtime. Must explain what a protocol adds beyond prompts and markdown: schemas, validators, artifact verification, state transfer, authority boundaries, provenance, reusable CLI commands, and machine-checkable continuation. No vibes.
PROMPT_MINIMALIST: Argues that ClisTa does not need to become a full protocol yet. The minimalist position is that one strong adversarial prompt plus one markdown transfer-state file already delivers most of the value. Must attack overbuilding, premature abstraction, protocol theater, and implementation drag.
PRODUCT_STRATEGIST: Evaluates whether ClisTa is a real product category. Must focus on user value, wedge, adoption path, narrative clarity, market confusion, and whether “ClisTa changes the shape of yes” is a strong enough product thesis.
PROTOCOL_SKEPTIC: Attacks the entire concept. Must challenge whether ClisTa is actually different from existing tools such as prompts, docs, project management, decision logs, evals, CI, agent frameworks, or governance checklists. Must identify failure modes where ClisTa could become bureaucracy, false confidence, or ceremonial state.
CLISTA STATE REQUIREMENTS: Each teammate must make their claims explicit and label them as one of:
CLAIM: something they assert is true
ASSUMPTION: something that must be true for their argument to hold
EVIDENCE_NEEDED: something that would need verification
OBJECTION: a concrete challenge to another side
CONCESSION: a point explicitly yielded
AMENDMENT: a change to the proposal required for support
RESIDUAL_RISK: a risk that remains even after rebuttal
BLOCKER: a condition that prevents the concept from advancing
DECISION_IMPACT: why the point should affect the final decision
